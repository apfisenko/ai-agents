## Итерация 5 — Учёт транзакций из текста (structured output) и отчёт о балансе

**Цель:** in-memory **TransactionStore** по `chat_id`; из **текстовых** сообщений пользователя — один вызов LLM с **structured output** (поля из раздела 6 [vision.md](vision.md)); сохранение распознанных транзакций; краткий ответ в чат. Команда **/balance** (или согласованный аналог) и простая сводка: баланс и при необходимости разбивка по категориям **из памяти** (без БД).

**Проверка:** несколько текстовых сообщений с тратами/доходами → строки в учёте; `/balance` показывает согласованный с контрактом отчёт; после рестарта процесса учёт пуст; ошибки API не светят секреты и сырые суммы в логах на INFO.

- [x] Модель данных транзакции (Pydantic/схема) и `TransactionStore`
- [x] Расширить `LlmClient` (или один соседний класс в отдельном файле — только если так проще): вызов с structured output для извлечения списка транзакций + опциональный пользовательский текст
- [x] Handler текстовых сообщений: запись в store → ответ
- [x] Handler `/balance` (и/или минимальная логика без лишнего NLU)
- [x] Обновить системный промпт под финансового советника и извлечение полей (согласовать с полями схемы)

Голосовые сообщения Telegram — итерация 8 ([vision про голос](vision.md)); прочее (видео, стикеры и т.д.) — по единой политике MVP (игнор или одна фраза «не поддерживается»).

**Проверка:** тестовое фото чека даёт минимум одну осмысленную транзакцию в store; при недоступности vision-модели — понятное сообщение пользователю; логи без полного OCR-текста на INFO.

## Итерация 8 — Транскрибация голосовых сообщений

**Цель:** пользователь отправляет **голосовое** сообщение Telegram → бот скачивает файл → один вызов к **мультимодальной LLM** с **аудио во входном сообщении** (подход «аудио в LLM», без отдельного ASR-сервиса) → **structured output** с той же схемой транзакций, что для **текста** ([vision раздел про LLM и сценарии](vision.md)) → запись в **TransactionStore**, обновление **ConversationStore** по правилу KISS (как после согласования с пользователем), ответ в чат. Отдельно менять «тип» бота (один пайплайн учёта после получения содержания реплики) не требуется.

**Зависимости:** поддержка аудио в chat completions выбранной **моделью** и **endpoint** (`base_url`). Если провайдер требует отдельную модель для audio — конфиг дополняется по [vision раздел моделей](vision.md); в `.env.example` — после реализации.

**Проверка:** тестовое голосовое с описанием траты даёт транзакции в store или осмысленный ответ модели пользователю; при недоступности аудио у модели/API — понятное короткое сообщение пользователю; логи без сырого содержания речи на INFO.

- [x] Handler голосовых (`Voice`): скачивание файла, единый выбранный способ упаковки во вход сообщения для OpenAI-compatible API (`input_audio` + base64, формат по MIME → ogg/mp3/wav и др.)
- [x] Метод в `LlmClient`: `extract_transactions_from_audio` — системный промпт + история (строками) + мультимодальный user; JSON как у VLM
- [x] Связка с `TransactionStore` / `ConversationStore`; `prompts/system.txt` — явное правило для голоса
- [x] `.env.example`, `ReadMe`: `LLM_AUDIO_MODEL`, `LLM_AUDIO_MAX_COMPLETION_TOKENS`
- [x] Ручная проверка: чеклист в ReadMe (раздел «OpenRouter и Ollama»); фактический прогон с голосовым на выбранной модели с audio input — на стороне оператора.
